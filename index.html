Mahall — Global Multi-Tenant E-commerce Infrastructure

Treating multi-vendor marketplace coordination as a distributed systems challenge rather than a monolithic storefront application.

The Operational Problem

Standard e-commerce checkouts force catalog lookups, inventory reservation, and payment captures to run as a single synchronous web transaction. As seller catalogs scale, slow payment gateways and concurrent inventory locks result in high checkout failure rates, database deadlocks, and split-shipment calculation errors.

The Architectural Solution

Mahall decouples order intake from processing. The backend accepts orders immediately at the edge and handles inventory reservation, payout allocation, and fulfillment asynchronously through an event-driven order orchestrator. Catalog updates are split from checkout storage, preventing ingestion spikes from degrading purchase funnels.

Key Technical Decisions

  • Decoupled Ingestion: Built distinct Catalog and Inventory microservices to prevent heavy writes from merchants from impacting checkouts.
  • Workflow Orchestrator: Implemented a distributed order coordinator running reservation checks and payment intents asynchronously.
  • Ledger-First Payouts: Designed seller payouts around a double-entry ledger database pattern, releasing funds only after third-party fulfillment endpoints issue delivery proofs.

Lead Source — Distributed Ingestion & Automation Control Plane

A multi-surface data collection and normalization pipeline separating web APIs from resource-heavy browser clusters.

The Operational Problem

Data enrichment workflows often rely on slow third-party REST APIs and complex browser automation (Playwright). Running these operations inside the primary HTTP thread pools causes server socket exhaustion, memory spikes, and severe request degradation for dashboard operators.

The Architectural Solution

Lead Source separates concerns into a Python API control plane, an asynchronous background worker pool, and a dedicated browser automation cluster. Incoming jobs are written to a Redis queue instantly, allowing workers to execute, normalize, and enrich data out-of-process.

Key Technical Decisions

  • Unified Adapter Pattern: Mapped static API sources, external HTTP scrapers, and dynamic browser automation onto a single backend interface contract.
  • Playwright Isolation: Deployed dynamic browser-assisted workers into containerized, resource-constrained sub-networks to prevent memory leaks from destabilizing the main application.
  • Telemetry Loops: Configured Prometheus metrics to monitor API requests, task completion latencies, and queue depth.

The Engine Room: Distributed Components

A catalog of custom-built backend engines, infrastructure adapters, and transactional tools. Each is built from scratch to solve specific latency, consistency, or scaling bottlenecks.

01

Transactional Outbox Relay

transactional-outbox

Problem: Committing business records (e.g. orders) and publishing events to external brokers in separate operations eventually causes failures where one step fails, leaving the system in an inconsistent state.

Solution: A Go library that writes domain models and integration events in the same SQLite transaction. A background dispatcher loop reads pending records and relays them to transport brokers with configurable exponential backoff and a dead-letter path.

Engine Detail: Rather than hiding failures in application logs, delivery events are committed directly to a queryable delivery ledger, making retry attempts auditability a first-class feature.

Go 1.24 SQLite zerolog Outbox Pattern
02

Custom Domain Validation Engine

custom-domain-automation

Problem: Standard custom domain setups in multi-tenant systems fail during onboarding handoffs because of unvalidated DNS records, causing SSL errors and manual troubleshooting.

Solution: A stateful automation service written in Go. It registers incoming custom hostnames, generates DNS TXT validation tokens, and runs an asynchronous polling routine to check record completion status.

Engine Detail: Features a fully decoupled DNS provider interface, isolating the status transition workflow from specific domain registrar configurations.

Go 1.25 Docker Compose DNS Provider Interface
03

Dynamic PostgreSQL Multitenant Engine

multitenant-engine

Problem: Shared-table data isolation risks information leakage and complicates customer-level data backups, while deploying separate database servers increases base infrastructure overhead.

Solution: A FastAPI backend that isolates tenants using dedicated PostgreSQL schemas. Tenant metadata is stored in a shared control registry under `public`, while tenant-owned business data is routed to `tenant_<slug>` tables.

Engine Detail: Schema provisioning is handled asynchronously. Workers fetch bootstrap instructions using PostgreSQL `FOR UPDATE SKIP LOCKED` locks to allow horizontal scaling.

Python 3.12 FastAPI asyncpg PostgreSQL 16
04

Redis-Backed Task Processor

background-task-processor

Problem: Full Celery integrations are too heavy for lightweight microservice architectures, yet built-in queues lack retry scheduling, backoff logic, and error persistence.

Solution: A FastAPI-linked task processor that uses Redis hashes for status storage, lists for pending jobs, and sorted sets for scheduled retries. It executes task callbacks out-of-process.

Engine Detail: Failed tasks write traceback summaries and attempt histories directly into the Redis hash metadata, preventing status loss.

Python 3.12 FastAPI Redis 7 redis-py
05

Worker-Backed Full-Text Search Index

go-fulltext-search

Problem: Standard database `LIKE` queries lack term stemming and relevance scoring, while deploying a full Elasticsearch cluster consumes excessive RAM and CPU resources.

Solution: A self-contained document search engine written in Go. It accepts document records over HTTP, processes index tokenization through a channel-based worker queue, and serves ranked searches.

Engine Detail: Ingestion calls are blocked until the inverted index is successfully updated, guaranteeing search availability matches write confirmation.

Go 1.24 Custom Inverted Index Docker Compose
06

Resilient Kafka Ingestion Pipeline

kafka-event-pipeline

Problem: Consumer process exceptions can lock up message queues or silently drop event payloads, leading to downstream state desynchronization.

Solution: A Python event ingress service built around the `confluent-kafka` client. It routes events across raw, retry, and dead-letter queues, handling bad payloads safely.

Engine Detail: Backoff delay values are encoded directly in the event wrapper, turning the message broker history into the absolute state authority for retry loops.

Python 3.12 FastAPI confluent-kafka Dead Letter Queues
07

Declarative Multisurface API Gateway

multisurface-api-gateway

Problem: Duplicating authentication, route-throttling, and health checking logic inside multiple service instances causes integration drift and security gaps.

Solution: A centralized reverse proxy written in Go. It intercepts incoming traffic, parses API key authorizations, and enforces route-level rate limits based on client identity.

Engine Detail: Active background health checks monitor downstream microservice endpoints, updating routing paths without server reloads.

Go 1.25 YAML Router Reverse Proxy Alpine
08

Go Metrics & Telemetry Adapter

observability-stack

Problem: Adding instrumentation as an afterthought results in tight coupling between business logic handlers and telemetry library clients.

Solution: A Go microservice pre-instrumented with Prometheus RED-style metrics and OpenTelemetry trace spans, wrapped in a clean local development compose stack.

Engine Detail: All tracing logic is decoupled using an injectable telemetry component. Features distinct endpoints for liveness (`/healthz`) and dependency readiness (`/readyz`).

Go 1.25 Prometheus Client OpenTelemetry

Engineering Philosophy

01

Simplicity Over Abstraction

Complexity increases operational costs. I design simple, readable architectures over clever, abstract layers. Code should be readable and straightforward to debug when a service fails under load.

02

Product-First Decisions

Technology exists to drive a business outcome. I write code with a deep awareness of product viability, billing quotas, and host cost efficiency.

03

Operational Telemetry

A system is only complete when it is observable. Structured logging, Prometheus metrics, liveness probes, and distributed traces should be built in from day one, not added late after an incident occurs.

04

Complete Ownership

Independent execution requires absolute accountability. I own my code from the first system diagram to live container orchestration and incident response.
