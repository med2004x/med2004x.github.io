Mahall — Multi-Tenant Commerce Operating System

Failure Mode
Standard checkout flows execute catalog lookups, inventory reservation, and payment capture in one synchronous transaction. Under concurrent load, slow payment gateways cause full table locks, and inventory checks deadlock with merchant catalog writes sharing the same DB connection pool.
System Design
The API accepts orders immediately at the edge and drops intent into Redis. An order orchestrator picks up async, fans out to decoupled Catalog and Inventory services, runs payment, and persists final state to PostgreSQL — never blocking the request thread on heavy I/O.
→Catalog and Inventory run as separate services so heavy merchant write bursts don't contend with checkout read paths.
→Order orchestrator coordinates reservation, payment intent, and payout allocation fully outside the HTTP request cycle.
→Seller payouts use a double-entry ledger pattern — funds release only after the fulfillment service confirms delivery proof.
→Storefront templates are tenant-scoped and served via Cloudflare R2 + CDN edge — no origin hit for static assets.

Lead Intelligence Platform github.com/med2004x/lead-source-architecture-case-study ↗

Lead Source — Three-Plane Lead Intelligence System

Failure Mode
Scraping jobs fan out across hundreds of source-location-keyword combinations and can't run synchronously from an API surface. Browser-assisted sources (Playwright) share heap with the HTTP process, causing OOM kills mid-run. Outreach messaging has completely different latency and compliance requirements than scraping — coupling them to one process model means messaging failures destabilize scraper throughput.
Architecture Decisions
Split into three independent planes per ADR-001: control (API + dashboard + scheduling), data (worker + adapters + browser cluster + external sources), and engagement (WhatsApp agent with its own SQLite state store). Redis is the execution backbone per ADR-002, keeping the API responsive while workers drain async. WhatsApp is a separate runtime boundary per ADR-003 — messaging failures don't touch scraper runs.
→Worker expands jobs into source-specific tasks routed through adapter implementations — static HTTP, API, and browser-assisted sources all conform to the same pipeline contract after extraction.
→Browser workers (Playwright remote server) run in a separate container with its own resource ceiling — a memory leak there doesn't reach the control plane API.
→Job lineage is persisted to PostgreSQL (jobs, runs, leads, exports, audit records) rather than reconstructed from logs — reruns and operator inspection work without bespoke tooling.
→WhatsApp agent consumes exported datasets via file handoff, maintains conversation state in SQLite, and runs its own webhook server — no coupling to scraper throughput or Redis queue health.

Backend Services

Eight backend services, each built to solve a specific consistency, isolation, or reliability problem. Real version numbers, real ports, real design decisions — all documented in public repos.

01 — :8093transactional-outbox ↗
Atomic Event Outbox Service

The dual-write problem: commit to the database, publish to the broker — one of them fails and your system is inconsistent. Orders and outbox entries are inserted in the same SQLite transaction, so domain state and integration intent commit or roll back together.

→Relay loop is config-driven — operators can also trigger a bounded dispatch batch through the API without waiting for the next poll interval.
→Every failed dispatch increments attempts, stores last_error, and schedules the next retry with exponential backoff. Status is queryable, not buried in logs.
→Dead-letter path is an explicit schema state (dead), not a separate queue — operators inspect it through the API without bespoke tooling.

02 — :8090custom-domain-automation ↗
DNS Verification & Domain Provisioning Service

Custom domain onboarding fails silently when DNS records haven't propagated — tenants see SSL errors with no explanation. This service treats verification and provisioning as a stateful workflow with a dedicated polling loop and explicit status transitions.

→DNS provider is behind an interface — swapping registrar integrations doesn't touch the state machine. Same flow works for sync demos and async real providers.
→Domains carry their verification token and target value in persisted state — the workflow survives restarts without replaying requests.
→/readyz reflects whether the poller loop is active, not just whether the HTTP process is alive — a real readiness distinction.

03 — :8094multitenant-engine ↗
PostgreSQL Schema-Per-Tenant Engine

Schema-per-tenant isolation: shared metadata in public, tenant-owned records in tenant_<slug> schemas — independently backable, auditable, and migratable per customer.

→Tenant creation is async — schema provisioning doesn't block control-plane writes or hit HTTP timeouts.
→Bootstrap jobs claimed with FOR UPDATE SKIP LOCKED — safe horizontal scaling without double-provisioning.
→Request middleware enforces auth globally and tenant-resolution only on tenant-scoped routes — no per-handler boilerplate.

04 — :8095background-task-processor ↗
Redis Retry & Dead-Letter Task Processor

Celery is too heavy for most of what I build. The gap is usually retry orchestration and failure visibility — teams enqueue work quickly but lose durable status tracking and a clean dead-letter story. Redis primitives fill that gap without a broker stack.

→Hashes for task state, lists for the pending queue, sorted sets for scheduled retries, dedicated list for dead-letter — each Redis primitive matches its access pattern exactly.
→Worker, retry scheduler, and cleanup loop run independently — bursts in one path don't stall the others.
→Task handlers are registry-driven — new task types add without touching queue plumbing. Failed tasks preserve last_error and attempt history, not silently re-enqueued.

05 — :8091go-fulltext-search ↗
Worker-Backed Full-Text Search Service

LIKE queries don't rank. Elasticsearch is RAM-heavy and operationally expensive for internal catalogs, back-office tools, or tenant-scoped knowledge bases that need ranked search without running a cluster.

→Writes are acknowledged only after a worker completes the index mutation — no false-success before search catches up.
→Raw documents live in a separate repository from the inverted index — ranking and token lookup stay isolated from document retrieval.
→Index ops move through a bounded channel (SEARCH_QUEUE_DEPTH) — backpressure is visible through GET /stats, not silently spawning goroutines.

06 — :8081kafka-event-pipeline ↗
Retryable Kafka Event Processing Pipeline

Event pipelines fail quietly — producers keep writing, consumers keep crashing, operators discover bad payloads after downstream state has already drifted. This pairs validated event intake with explicit retry state and an inspectable dead-letter topic.

→Retry timing encoded in the message envelope — backoff state travels with the event and is readable from the topic history without separate tooling.
→Transport is config-driven: embedded asyncio for deterministic local dev, kafka mode switches to real confluent-kafka adapter. Same routing logic either way.
→4 explicit topic states: raw → retry → processed → dead. Routing logic is isolated from the transport adapter so it's testable without a broker process.

07 — :8088multisurface-api-gateway ↗
Route-Driven Multi-Surface API Gateway

When product traffic splits across storefront, merchant, and back-office upstreams, each service tends to reinvent auth and throttling independently. This gateway centralizes those controls while keeping route behavior declarative through committed YAML config.

→API keys referenced by symbolic IDs in route config, resolved from env at startup — secrets stay out of source control, key rotation doesn't require a recompile.
→Rate limiting enforced per route and per caller key — noisy traffic is isolated to the surface generating it, not pooled globally.
→Upstream readiness computed from active health probes, not assumed from process status. Admin endpoints expose routing and probe state without shelling into containers.

08 — :8096observability-stack ↗
Prometheus & OpenTelemetry Observability Stack

Teams add metrics and tracing late, after incidents prove that logs alone are insufficient. The hard part isn't adding one library call — it's a coherent telemetry path: RED metrics, trace emission, scrape config, and split health probes all lined up from day one.

→Telemetry wrapped in an injectable Observability component — handlers stay focused on request behavior, not instrumentation wiring.
→Separate /healthz (process alive) and /readyz (dependencies configured) — a real liveness/readiness distinction that Kubernetes and load balancers can act on differently.
→Prometheus scrape config committed alongside the service — scrape targets stay versioned with the code they observe. OTel spans emit to stdout locally for zero-dependency trace inspection.

Engineering Principles
01
Explicit Systems Over Clever Abstractions
Abstractions cost debugging time. When a service fails at 2am, readability matters more than cleverness. I add layers when the problem requires it — not as a default.
02
Infrastructure Cost Is Part of the Design
I run my own infrastructure. Every design decision has a hosting cost attached. I treat that as a constraint from day one — not an optimization for later.
03
Observability Before Incidents
Logs alone aren't enough. Structured logs, RED metrics, and split liveness/readiness probes get built in first. Retrofitting telemetry after an incident is avoidable.
04
End-to-End Product Ownership
I take projects from schema design through deployment and incident response. No handoffs to a separate ops layer, no "that's DevOps's problem."
